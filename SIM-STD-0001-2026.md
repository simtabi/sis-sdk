## Foreword

This specification defines how Simtabi names things. It is modelled on the practices of the
institutions that have had to solve this problem at scale and could not afford to get it wrong: the
IEEE (MAC/OUI), IANA (IP address space), the IETF (RFC numbering), ISO (standards numbering), GS1
(trade identifiers), NASA (COSPAR), Apple, and IBM.

Three properties are borrowed and are non-negotiable here:

1. **Permanence.** An identifier that has been *commissioned* is fixed forever. It is never edited,
   never reclaimed, never reissued. An IEEE OUI assigned in 1985 is still that company's OUI today,
   even if the company is dead.
2. **Additive evolution.** New editions may *add* classes, widen serials, and define new forms. They
   may never invalidate an identifier issued under a previous edition. IPv6 did not break IPv4;
   SIS/2 will not break SIS/1.
3. **Separation of identity from state.** The identifier says *which thing*. It never says anything
   about the thing's current condition, owner, price, status, or employment arrangement. Those
   change. Identifiers do not.

### Conformance language

**MUST** / **MUST NOT** are absolute. **SHOULD** admits an exception where the reason is recorded.
**MAY** is optional.

## 1. Scope

This specification applies to every persistent entity Simtabi creates, receives, or transacts
against: clients, people, vendors, projects, contracts, invoices, products, services, releases,
components, assets, documents, standards, decisions, environments, incidents, and tickets.

It does **not** apply to:

- Ephemeral objects (session tokens, cache keys, log lines).
- Identifiers owned by third parties (a client's PO number, a vendor's SKU). Those are *recorded*
  against a Simtabi identifier, never substituted for one.

---

## 2. Grammar

### 2.1 Canonical form

An identifier conforming to SIS/1 takes one of exactly two forms.

**Form G — global:** `SIM-{CLASS}-{SERIAL}-{CHECK}` — e.g. `SIM-PRS-100001-FA`
**Form S — scoped:** `SIM-{CLASS}-{SCOPE}-{SERIAL}-{CHECK}` — e.g. `SIM-INV-ADIQ-000001-VY`

### 2.2 Segments

| Segment | Definition | Mutable? |
| --- | --- | --- |
| `SIM` | Issuer prefix. Simtabi's equivalent of an IEEE OUI. | **Never.** Fixed for the life of the company. |
| `CLASS` | 3 or 4 uppercase letters. Names the *kind* of thing. | **Never**, once commissioned. |
| `SCOPE` | 4–6 chars, `[A-Z][A-Z0-9]{3,5}`. The owning client's mnemonic. Present only in Form S. | **Never**, once commissioned. A project cannot move clients. |
| `SERIAL` | 6 to 9 digits, zero-padded. | **Never**, once commissioned. |
| `CHECK` | 2 alphanumeric characters, derived (§4). | **Never** — it is a function of the rest. |

**Every segment of a commissioned identifier is immutable.** State, ownership, description, price,
and status live in the register (§6), never in the identifier.

### 2.3 Formal grammar (ABNF)

```abnf
identifier   = global / scoped
global       = "SIM-" class "-" serial "-" check
scoped       = "SIM-" class "-" scope "-" serial "-" check

class        = 3*4ALPHA
scope        = ALPHA 3*5(ALPHA / DIGIT)
serial       = 6*9DIGIT
check        = 2(ALPHA / DIGIT)
```

Regular expressions, normative:

```
Form G:  ^SIM-[A-Z]{3,4}-[0-9]{6,9}-[0-9A-Z]{2}$
Form S:  ^SIM-[A-Z]{3,4}-[A-Z][A-Z0-9]{3,5}-[0-9]{6,9}-[0-9A-Z]{2}$
```

### 2.4 Case and separators

Identifiers **MUST** be stored, transmitted, and printed uppercase with hyphen separators.
Comparison **MUST** be case-insensitive and **MUST** ignore separators, so `sim-prs-100001-fa` and
`SIMPRS100001FA` both resolve to `SIM-PRS-100001-FA`.

## 3. Class register

A class is a namespace. `SIM-INV-…` can never be confused with `SIM-PRS-…`. **Class codes are
allocated by this specification and are never reassigned.** If a class is retired, its code is
retired with it.

> **Note on `STD` (correction — Annex C.2).** A `STD` register identifier is an ordinary Form G
> identifier with a **6-digit serial starting at `000001`** (e.g. `SIM-STD-000001-XX`, Annex A). The
> serial obeys the §2.3 grammar (`6*9DIGIT`) like every other class; the low start simply reflects
> that Simtabi will issue few standards. The **designation** of a standard — `SIM-STD-0001:2026` — is
> an ISO-style *standard number* (number `0001`, edition year `2026`), **not** a SIS identifier: it
> carries no check characters and predates this grammar. The two are different artefacts and are not
> interchangeable.

### 3.6 Reserved class codes

**MUST NOT** be allocated: `SIM`, `STD` (except as defined in §3.4), `TST`, `TMP`, `XXX`, `NIL`,
`VOI`, `SYS`, `ADM`, `ROO`.

### 3.7 Controlled vocabularies (subtypes)

Some classes carry a **subtype**: an *attribute* recorded in the register's `subtype` column, **never**
a segment of the identifier. A laptop's identifier is `SIM-AST-100001-XX`; *that it is a laptop* is a
column. Buying a new kind of thing never requires a new class code.

| Class | Permitted subtypes |
| --- | --- |
| `AST` Asset | `LAP` · `MON` · `PHN` · `SRV` · `LIC` · `DOM` · `KEY` · `MSC` |
| `DOC` Document | `ICA` · `MSA` · `SOW` · `NDA` · `CHG` · `DPA` · `IPA` · `EMP` · `QUO` |
| `PRS` Person | `ENG` · `DES` · `PM` · `OPS` · `BIZ` · `EXE` |
| `DPT` Department | `ENG` · `DES` · `OPS` · `BIZ` · `FIN` · `EXE` |

A class **not** listed here carries **no** subtype: its `subtype` column **MUST** be null. A
vocabulary **MAY** be extended in a new edition; an entry, once used, **MUST NOT** be removed or
redefined. **A person's subtype is a role, and a role changes** — it is a mutable column, unlike
everything in the identifier itself.

## 4. Check characters

### 4.1 Algorithm

The check is **ISO 7064 MOD 1271-36**, the pure double-character system over the alphabet
`0123456789ABCDEFGHIJKLMNOPQRSTUVWXYZ`.

```
M = 1271, r = 36
P = 0
for each character c of the payload:
    P = ((P + value(c)) * r) mod M
P = (P * r) mod M
check = (M + 1 - P) mod M
c1 = ALPHABET[check div r]
c2 = ALPHABET[check mod r]
```

The payload is the identifier with separators removed and the check characters excluded.

### 4.2 Trade study: why two characters and not one

Three algorithms were implemented and tested **exhaustively** over every identifier shape in §3.

| Algorithm | Check size | Substitution | Adjacent transposition | Verdict |
| --- | --- | --- | --- | --- |
| ISO 7064 **MOD 37,36** | 1 char | 100% | **99.06%** | **Rejected** |
| ISO 7064 **MOD 97-10** (IBAN) | 2 digits | **99.49%** | 100% | **Rejected** |
| ISO 7064 **MOD 1271-36** | 2 chars | **100%** | **100%** | **Adopted** |

MOD 37,36 fails adjacent transpositions (`SIM-PRS-100001` and `SIM-PRS-100010` collide). MOD 97-10
expands each letter into two digits, so a single substituted letter can cancel (missed 27/5,320).
MOD 1271-36 detects 100% of single-character substitution, adjacent transposition, jump transposition
(`aXb`→`bXa`), and twin errors (`aa`→`bb`).

> **Do not re-propose MOD 37,36 or MOD 97-10.** Both were measured and failed. A future edition
> proposing a new algorithm **MUST** publish comparable exhaustive figures and **MUST NOT** change
> the algorithm for identifiers already issued (§10).

### 4.3 Requirement

Every SIS/1 identifier **MUST** carry check characters. An identifier presented without a valid check
**MUST** be rejected, not repaired.

## 5. Mnemonic aliases

A client carries **both** a canonical identifier (`SIM-CLT-100001-9O`, machine-facing, immutable) and
a mnemonic alias (`ADIQ`, human-facing, unique, **also immutable once commissioned**). This is the
DNS pattern. The alias appears as the `SCOPE` segment of every Form S identifier belonging to that
client, which is **why it can never change** — it is embedded in every invoice, SOW, and ticket ever
issued against them.

### 5.1 Alias grammar

```
[A-Z][A-Z0-9]{3,5}          4 to 6 characters
```

### 5.2 Derivation

Aliases are derived by algorithm and confirmed by a human. The scarce resource is *memorable* codes,
so the algorithm **widens before it mangles**: it tries every 4-character candidate, then 5, then 6,
and only resorts to a numeric discriminator when a memorable code is genuinely impossible. `ACMX`
reads like Acme; `ACME2` reads like a database error.

Derivation, in order: (1) transliterate accents; (2) expand `&`→`AND`, strip non-`A-Z0-9`; (3) strip
legal suffixes (LLC, Inc, Ltd, Corp, GmbH, …); (4) set aside generic words (Holdings, Group,
Solutions, Technologies, …) only if something distinctive survives; (5) propose ranked candidates
(head + distinctive tail; truncation; initials padded from the first word; consonant skeleton; a
re-admitted generic; first-two + last-two); (6) pad short names with `X`.

| Legal name | Alias | Why |
| --- | --- | --- |
| AdelsaIQ LLC | `ADIQ` | head + distinctive tail |
| Acme Corp | `ACME` | straight truncation |
| Acme Inc | `ACMX` | `ACME` taken; widened |
| Acme Industries Ltd | `AICM` | initials + first word |
| Café Solutions GmbH | `CAFE` | transliterated; generic word set aside |
| Zed & Partners LLP | `ZEDX` | `&` expanded, then padded |
| Northwind Traders | `NORS` | generic word re-admitted to break the tie |
| Northwind Technologies | `NOND` | same name, different tail |
| Simtabi LLC | `SIBI` | `SIMT` is reserved |

### 5.3 Reserved aliases

`SIMT` `PROS` `TEST` `NULL` `VOID` `TEMP` `DEMO` `NONE` `ADMIN` `ROOT` `SYST`

Internal Simtabi work uses `SIMT`. Prospects sit under `PROS` until they sign. **Nothing under `PROS`
is ever invoiced.**

## 6. Lifecycle

### 6.1 States

| State | Meaning | Releasable? |
| --- | --- | --- |
| `RESERVED` | Allocated but not yet in use. | **Yes** |
| `COMMISSIONED` | In use. **Immutable forever.** | **No. Never.** |
| `SUSPENDED` | Temporarily inactive. Still immutable, still owned. | No |
| `DECOMMISSIONED` | Retired. The thing is gone; the identifier is not. | No — terminal |
| `VOID` | Issued in error, never used. | No — terminal |

### 6.2 Transitions

`RESERVED → {COMMISSIONED, VOID}`; `COMMISSIONED → {SUSPENDED, DECOMMISSIONED}`;
`SUSPENDED → {COMMISSIONED, DECOMMISSIONED}`; `DECOMMISSIONED` and `VOID` are terminal.

### 6.3 Invariants

1. **No path returns to `RESERVED`.**
2. **A commissioned identifier MUST NEVER be released, reused, or reissued.** Not on decommissioning,
   client loss, or employee departure. Not ever.
3. **A commissioned identifier MUST NEVER become `VOID`.** An error in a commissioned record is
   corrected by supersession (§8), not by voiding.
4. `DECOMMISSIONED` and `VOID` are terminal.
5. `SUSPENDED` may be restored to `COMMISSIONED`.

### 6.4 Locking

Commissioning **locks** the identifier. A locked identifier's segments cannot be edited by any means,
by any actor, including an administrator. Systems implementing SIS/1 **MUST** enforce this at the
storage layer — in SQL, a trigger or a `CHECK` constraint, not merely application code. The mental
model is a MAC address burned into a NIC.

### 6.5 Reservation

A reservation **MUST** record who reserved it, when, and why; **SHOULD** carry an expiry (an unclaimed
reservation past expiry is released back to the pool); and **MUST** be released or voided explicitly —
it never lapses silently.

## 7. Products, services, and releases

A **product** (`PRD`) is something Simtabi owns and can sell more than once; a **service** (`SVC`) is
a repeatable engagement shape; a **component** (`CMP`) is internal reusable kit. Each carries a
canonical identifier and a mnemonic alias.

### 7.2 Release

A **release** is an immutable, shipped version of a product. Once published it is never edited; a
defect is fixed by shipping a *new* release.

```
{PRODUCT_ALIAS}-{MAJOR}.{MINOR}.{PATCH}[-{PRERELEASE}][+{BUILD}]
MALISA-1.4.2
MALISA-2.0.0-rc.1
MALISA-1.4.3+20260712.a91f2c
```

Precedence follows Semantic Versioning 2.0.0: build metadata is ignored for ordering; a pre-release
sorts below its release. Every release also carries a canonical identifier (`SIM-REL-100001-…`). A
published release **MUST NOT** be edited, re-tagged, or replaced in place; a withdrawn release is
marked `DECOMMISSIONED` and its version number is never reused; version numbers within a product are
strictly increasing.

### 7.3 Client deliverables are not products

Work built *for* a client under a SOW is that client's Work Product, not a `PRD`.

## 8. Correction and supersession

Identifiers are never edited. Neither are the records that carry them. A wrong invoice is credited
(`CRN`) and reissued under a new `INV`; a wrong document gets a new version with `superseded_by` set;
a wrong release is superseded by a new one; a wrong standard is replaced by a new edition. **The chain
of supersession is the audit trail. Editing history destroys it.**

## 9. The register

One table. One source of truth. If it is not in the register, it does not exist.

| Column | Type | Mutable? |
| --- | --- | --- |
| `identifier` | text, primary key | **Never** |
| `class` | text | **Never** |
| `scope` | text, nullable | **Never** |
| `serial` | bigint | **Never** |
| `alias` | text, nullable, unique | **Never** once commissioned |
| `state` | enum (§6.1) | Yes, per §6.2 only |
| `description` | text | Yes |
| `owner` | identifier → `PRS` | Yes |
| `subtype` | text, nullable | Yes (an attribute, §3.7) |
| `reserved_at` / `reserved_by` / `reserved_reason` | | Set once |
| `commissioned_at` | timestamp | Set once |
| `decommissioned_at` | timestamp, nullable | Set once |
| `superseded_by` | identifier, nullable | Set once |
| `spec_edition` | text (`SIS/1`) | **Never** |

Requirements: unique index on `identifier`; unique index on `alias` where not null; serial issuance
**MUST** be transactional (two callers **MUST NOT** receive the same serial); immutability of the
frozen columns **MUST** be enforced in the database; `spec_edition` is stamped at issue.

## 10. Evolution of this specification

SIS follows the ISO convention: a number, an edition, and a year (`SIM-STD-0001:2026`).

**Permitted (additive):** allocating new class codes; widening the class-code length (3→4 letters, so that
every previously-issued 3-letter identifier stays valid while 4-letter codes such as `CUST` become available
to custom registers); widening the serial range (6→9 digits); defining new forms, aliases, lifecycle states;
adding classes of check.

**Forbidden (destructive):** invalidating an identifier issued under a previous edition, ever;
reassigning a retired class code; narrowing a serial range; changing the check algorithm for
identifiers already issued (a new algorithm applies only to new issues and requires a new class of
check recorded in `spec_edition`).

---

## 11. Conformance

An implementation conforms to SIS/1 if it: (1) generates identifiers matching §2.3 exactly;
(2) computes and verifies check characters per §4.1; (3) enforces every lifecycle invariant in §6.3;
(4) enforces immutability of commissioned identifiers at the storage layer (§6.4); (5) never reuses a
serial within a class and scope; (6) stamps `spec_edition` on every issue.

The reference implementation is the **`simtabi/sis`** package (with its Laravel binding
`laranail/sis-wrapper`). Where the package and this document disagree, **this document is normative**
and the package is defective.
